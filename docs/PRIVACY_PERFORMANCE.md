# Asteria privacy and performance architecture

This document describes protocol version 5. It is normative for the current
pre-release network. Version 5 deliberately has no migration path from version
4: reset both CometBFT data and every Asteria `redb` database before starting it.

## Execution model

- CometBFT 0.38 provides ordering and finality through ABCI++.
- Public GTC orders normally remain in the continuous limit order book. At the
  end of a block containing private submissions, eligible non-reduce-only GTC
  liquidity for those markets is moved into a committed per-height escrow.
  Its reference mark and original price-time sequence are frozen.
- Private orders are one-batch IOC or FOK orders. Side, price, quantity,
  leverage, reduce-only state, and client identifier are encrypted.
- A ciphertext accepted and committed in block `H` receives 3-of-4 validator
  decryption and beacon shares in the vote extensions for `H+1`. The first
  system transaction in `H+2` verifies those shares and executes the batch.
  Validators never share ciphertexts from an uncommitted proposal.
- Admission at `H` requires
  `available_margin >= private_order_fee + 100`. The non-refundable fee is
  deducted first, then all available margin remaining after the fee is moved
  into `reserved_margin` as the bond. An outer signed transaction that expires
  before admission is rejected without a fee or bond. A committed batch has no
  silent timeout cleanup and must reach a valid mandatory `H+2` finalization.
- Orders submitted to the live book during `H+1` cannot join the `H` batch.
  The `H` batch clears only against its escrow and frozen reference mark. Any
  unfilled public remainder is restored with its original sequence after the
  batch, then revalidated against the current mark-price risk band.
- Private orders and eligible public GTC liquidity clear at one deterministic
  fixed-point price. The clearing rule maximizes volume, minimizes imbalance,
  minimizes reference-price distance, and finally selects the lower price.
  Boundary liquidity is allocated by deterministic integer pro-rata.
- Markets clear in parallel. Account, order-book, trade, and event mutations
  are merged in stable market and order-key order before the state commitment is
  calculated.

## Private-order trust model

The private-order key is generated by the Zcash Foundation FROST Ristretto255
two-round DKG with threshold 3 and four participants. Every ceremony is bound
to the chain domain, epoch, and a fresh random 32-byte ceremony identifier. No
code path reconstructs the group secret. Round-one DKG messages require
authenticated consistent broadcast; round-two messages require authenticated
confidential point-to-point channels. The localnet key generator runs those
channels inside one offline process and writes four separately permissioned
shares plus the canonical ceremony record.

`InitChain` enables private orders only when the CometBFT genesis set contains
exactly four unique Ed25519 validators with equal positive voting power, vote
extensions start at height 1, and the four derived validator addresses exactly
match the threshold-share bindings. Consequently every commit with more than
two-thirds voting power contains at least three decryption-share identities.

Orders use threshold Ristretto255 DH, HKDF-SHA256, and
XChaCha20-Poly1305. Decryption shares carry Chaum-Pedersen DLEQ proofs. A bundle
may omit or contain one invalid validator share, but three distinct valid
shares are mandatory. An invalid ciphertext or plaintext invalidates only that
order; a malformed bundle or invalid share proof invalidates the system
transaction.

The committed `H` application hash, chain, height, and epoch key derive a
hash-to-Ristretto beacon base only after the order set is final. Each validator
publishes its Shamir share of that point with a Chaum-Pedersen proof. Any valid
3-of-4 subset interpolates the same unique group point; proof randomness and
the proposer's choice of valid subset cannot change the result. The verified
threshold beacon, chain, height, and market domain seed pro-rata remainder
priority. Traders choose order identifiers before the beacon is known, so
rerandomizing ciphertexts cannot target a known allocation seed.

This removes proposer/order-ID grinding after the cutoff. It does not prevent
inclusion censorship before block `H` commits, and three colluding threshold
validators can always violate the threshold privacy assumption. Fee payer,
market, ciphertext size, submission height, and timing remain public metadata.

## Shielded isolated margin

Shielded state has note commitments, nullifiers, a depth-32 append-only Merkle
tree with recent-root history, authority-backed deposits, collateral and
position conservation, public fees, and isolated-margin policy checks. Public
spend inputs contain only the chain/ledger domains, anchor, nullifiers, output
commitments, policy domain, and fee. Consumed notes, leaf indices, Merkle paths,
and output openings belong to the proof witness.

The consensus integration is development-only. Its transparent proof bytes
still reveal note openings and Merkle paths to validators and chain observers;
moving them out of the public statement defines the production verifier
boundary but does not make the development backend zero knowledge. It must be
enabled explicitly in genesis.
Production profile types reject development verifiers, but an audited
production zero-knowledge circuit and verifier are not included yet. Therefore
the current network must not claim private positions or production shielded
margin. Cross-margin is intentionally unsupported because it would couple
otherwise isolated private solvency proofs.

## State and accounting

Consensus state is split into entity leaves in a SHA-256 Jellyfish Merkle Tree
and persisted atomically in `redb`. Stale JMT nodes and value tombstones are
removed in the same transaction, so the database retains the current
authenticated tree rather than pretending to serve historical proofs.
Immutable `imbl::OrdMap` and `imbl::Vector` values make proposal snapshots and
rollback structure-sharing operations.

The mandatory `H+2` system transaction validates the threshold bundle before
releasing each private-order bond. Bond release then precedes per-order
decryption results and frozen-mark risk validation. With a valid bundle, an
invalid ciphertext or plaintext invalidates only that order; successful batch
completion removes the pending orders, bond records, committed app-hash anchor,
and frozen snapshot together. The entire state transition is atomic. If the
bundle or system transaction fails, neither any earlier bond release nor cleanup
is committed, and the batch remains intact for deterministic retry.

Funding updates are O(1): each market stores a cumulative funding index, and a
position settles only when touched. Settled payments pass through a dedicated
funding pool, never the insurance fund. Auditing checks:

```text
funding_pool + sum(position_qty * (market_index - position_index)) == 0
```

The pool is not added to asset conservation because account equity already
includes pending funding. The audit independently checks balanced open
interest, order reservations, shielded backing, fees, insurance, event hashes,
and total credited collateral. Order admission applies a mark-price band and
reserves initial margin, taker fees, and worst immediate loss relative to mark.
Post-fill risk is checked atomically. Liquidation bad debt uses non-negative
insurance first, then the fee vault, then a bounded deterministic social-loss
backstop; an actually insolvent transition rolls back instead of committing a
negative protocol reserve.

## Resource bounds

- User transactions: 256 KiB.
- Private payload plaintext: fixed 1024-byte padding.
- Pending private orders: protocol bounded per batch and one per fee payer.
- Shielded inputs and outputs: 64 each; proof: 1 MiB.
- Verified-transaction cache: LRU bounded by both entry count and encoded bytes.
- Frozen private-batch state, transaction caches, and retained event history
  are protocol bounded.

Stateless transaction verification and independent market auctions use Rayon.
ABCI state transitions remain serialized, while Info, Query, proposal preview,
and vote-extension reads share a read lock.

## Deferred cryptographic work

Before a production launch, integrate and audit a real shielded-margin circuit,
wallet viewing encryption, setup/key rotation, and recovery procedures. A
reviewed batch-threshold-encryption construction may later reduce the current
O(validators x ciphertexts) vote-extension traffic; it must not be replaced by
an ad hoc aggregation scheme.
